Key takeaways

  1. Different table types for different access patterns. There is no single "fastest" option. Hybrid dominates point lookups; Interactive dominates group lookups.
  2. Concurrency reveals what serial tests hide. Phase 1 showed Hybrid and Interactive nearly tied. Phase 2 showed Hybrid scaling to 421 QPS while Interactive peaked at 149 QPS for point lookups.
  3. Hybrid is terrible for bulk reads. At c=1, Hybrid returns 12.5k rows in 2.4 s — worst of all variants. The row-store secondary index walk is fundamentally slower than columnar scans.
  4. Standard tables saturate quickly. On a single XSMALL cluster, throughput caps at ~10 QPS regardless of how many clients you throw at it. All additional concurrency becomes queue wait.
  5. Search Optimization has diminishing returns here. SOS improved point-lookup latency by ~10% but didn't meaningfully help group lookups. The data is already well-suited to micropartition pruning.
  6. Compilation time matters. Interactive compiles in 16 ms vs 80-120 ms for Standard. At sub-second latency targets, this is a major factor.
  7. Network overhead is the floor. ~150-200 ms of every query is client-side overhead. To go below 200 ms end-to-end, you'd need a co-located client (e.g. SPCS service function).
