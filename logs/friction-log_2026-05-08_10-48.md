## Friction Points

### F1 — `ors_control_app` image tag mismatch between `image-versions.env` and `ors_control_app_service.yaml`

- **Where:** Step 6, executing module `01_core_infra.sql`.
- **Symptom:** `CREATE SERVICE ors_control_app` failed with
  `397012 (23001) ... Image /openrouteservice_app/core/image_repository/ors_control_app:v1.0.146 not found`.
- **Root cause:** `openrouteservice_app/image-versions.env` has `ORS_CONTROL_APP_TAG=v1.0.141`, but the YAML file `services/ors_control_app/ors_control_app_service.yaml` was hard-coded to `v1.0.146`. The build step pushed `v1.0.141` (the env value, which the skill calls "the single source of truth"), but the CREATE SERVICE statement used the YAML, which referenced a tag that was never built.
- **Resolution applied in this run:** Edited `ors_control_app_service.yaml` to reference `v1.0.141`, re-uploaded the YAML to `@ORS_SPCS_STAGE/services/ors_control_app/`, and ran the missing `CREATE SERVICE ors_control_app` statement individually. Modules 02–06 then ran cleanly.
- **Prevention recommendation:**
  1. Add an explicit pre-deployment validation step (the skill already mentions this in passing in Step 6 — promote it to a hard, automated check). Concretely, add a small bash check before module 01 runs:
     ```bash
     ENV_TAG=$(grep '^ORS_CONTROL_APP_TAG=' openrouteservice_app/image-versions.env | cut -d= -f2)
     YAML_TAG=$(grep -oE 'ors_control_app:v[0-9.]+' openrouteservice_app/services/ors_control_app/ors_control_app_service.yaml | cut -d: -f2)
     [ "$ENV_TAG" = "$YAML_TAG" ] || { echo "ERROR: env tag $ENV_TAG != yaml tag $YAML_TAG"; exit 1; }
     ```
  2. Even better: have `image-versions.env` be the only place tags live, and make the YAML `image:` line generated/templated rather than hand-edited. A simple `envsubst` step in the build script would eliminate the drift.
  3. Same drift could affect the other 4 service YAMLs — extend the check to all 5 services.

### F3 — `snow sql -f .../deploy-agent.sql` final verification query failed (cosmetic)

- **Where:** Step 8.5 (Routing Agent).
- **Symptom:** The agent and three procedures created successfully (last status before the error: `Agent ROUTING_AGENT successfully created.`), but the trailing verification `SELECT ... FROM INFORMATION_SCHEMA.PROCEDURES` failed with
  `090105 (22000) ... This session does not have a current database. Call 'USE DATABASE'`.
- **Root cause:** The verification query in `deploy-agent.sql` uses an unqualified `INFORMATION_SCHEMA.PROCEDURES` reference, but `snow sql -f` does not carry session DB context unless explicitly set.
- **Resolution applied:** Treated as cosmetic — the agent and procedures were verifiably created (agent shows in `SHOW AGENTS`).
- **Prevention recommendation:** In `references/deploy-agent.sql`, prepend `USE DATABASE FLEET_INTELLIGENCE; USE SCHEMA ROUTING_AGENT;` before the final verification block, OR replace `INFORMATION_SCHEMA.PROCEDURES` with `FLEET_INTELLIGENCE.INFORMATION_SCHEMA.PROCEDURES`. The latter is preferred per AGENTS.md "Always use fully qualified object names."

### F4 — Background bash + `tail -3` on `docker push` swallows the final "DONE" sentinel

- **Where:** Step 5a (openrouteservice push).
- **Symptom:** When piping `docker push ... 2>&1 | tail -5`, the command's `\r` progress writes overwrite each other and the final `digest:` line, plus the trailing `&& echo "PUSHING ..."` sentinel, sometimes don't appear in `bash_output` even after the push completes. Required cross-checking with `snow spcs image-repository list-images` to confirm completion.
- **Resolution:** Verified push completion via `list-images` instead of trusting the captured stdout.
- **Prevention recommendation:** The skill already documents this in `references/build-images.md` ("Push progress output uses carriage returns... do not assume a push is stuck"). Consider adding a small explicit recommendation: "After every push, immediately run `snow spcs image-repository list-images` and grep for the just-pushed tag to confirm. Do not rely on captured stdout."
