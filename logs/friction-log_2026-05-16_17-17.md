## Friction points

### F1. `INFORMATION_SCHEMA.QUERY_HISTORY_BY_WAREHOUSE` is not in `INFORMATION_SCHEMA`

**What happened:** First attempt `SELECT * FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY_BY_WAREHOUSE('ROUTING_ANALYTICS'))` failed with `Invalid identifier`. The function lives at `SNOWFLAKE.INFORMATION_SCHEMA.QUERY_HISTORY_BY_WAREHOUSE` and takes named arguments `WAREHOUSE_NAME =>` and `RESULT_LIMIT =>`.

**Resolution this run:** Switched to `TABLE(SNOWFLAKE.INFORMATION_SCHEMA.QUERY_HISTORY_BY_WAREHOUSE(WAREHOUSE_NAME => '...', RESULT_LIMIT => 50))` and the query worked.

**Recommendation:** Add a tiny SQL snippet to `references/troubleshooting.md` (or wherever common-helpers live) showing the canonical "is anything running on warehouse X" probe. Current skills use a mix of phrasings; standardising it would prevent future trial-and-error.

### F2. Silent error swallow in the original auto-resize block was the actual root cause

**What happened:** The 2026-05-15 stranding at X-Large was caused by:
```sql
EXCEPTION WHEN OTHER THEN
    original_wh_size := NULL;   -- silently swallows; restore becomes a no-op
END;
```
combined with a restore guard `IF (original_wh_size IS NOT NULL)`. Any failure inside the size-capture block guaranteed the restore would be skipped.

**Resolution this run:** Replaced with a non-NULL default (`'SMALL'`) plus a `did_bump BOOLEAN` flag that strictly controls whether the restore runs. Also reject LARGE/X-LARGE/etc. as a captured "original" (treat as drift from a prior un-restored run and shrink to steady-state instead).

**Recommendation (longer-term):** Add a code-review checklist item / lint rule for any `EXCEPTION WHEN OTHER THEN` block that mutates a control variable used by a later restore step. The pattern of "swallow + null-guard restore" is the precise anti-pattern that bit us here. The new structure (`did_bump` separate from `original_wh_size`) avoids it.

### F3. Steady-state size is hard-coded in two places (5 occurrences total)

**What happened:** The new code embeds `'SMALL'` as a literal in:
- `05_matrix_pipeline.sql` line ~1128 (default for `original_wh_size`)
- `05_matrix_pipeline.sql` size-allowlist `('X-SMALL','XSMALL','SMALL','MEDIUM')`
- `04_service_lifecycle.sql` `RECONCILE_WAREHOUSE_SIZE` body (default + allowlist + final ALTER)

If we ever change the steady-state, we have to update all of them.

**Resolution this run:** None — kept it simple and consistent with how thresholds are hardcoded elsewhere in the matrix pipeline.

**Recommendation:** Future enhancement — move the steady-state size and the bump thresholds (5000 / 25000 hex) into a tiny `OPENROUTESERVICE_APP.CORE.CONFIG` table or scalar function. Out of scope for this fix.
