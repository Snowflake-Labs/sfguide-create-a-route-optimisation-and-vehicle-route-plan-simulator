# Friction Log - Build Routing Solution + Cleanup
Date: 2026-04-12 20:57 - 21:45 PT

## Context
Attempted full build-routing-solution from scratch, got stuck on Step 5 (docker push openrouteservice taking 15+ min with no visible progress), then ran full cleanup.

## Friction Points (Build)

### F1: Docker push output invisible with `tail -5`
- **Step:** 5 - Build and push container images
- **Severity:** Medium (workflow friction)
- **Description:** The build-images.md reference uses `tail -5` to capture build/push output, but `docker push` uses carriage returns (\r) for progress updates. Combined with `tail -5`, only the initial push line is ever visible. The agent cannot confirm push completion until the entire chain finishes.
- **Suggestion:** Either (1) remove `tail -5` from push commands and accept verbose output, or (2) add explicit verification after each push with `docker manifest inspect`, or (3) add a note that push progress won't be visible and to wait for the chain to complete.

### F2: Snow CLI version warning noise
- **Step:** 4 - Upload configuration files
- **Severity:** Low
- **Description:** Every `snow` CLI command emits `UserWarning: New version of Snowflake CLI available. Newest: 3.16.0, current: 3.9.0` which clutters output. Not a real problem but adds noise when parsing results.
- **Suggestion:** Add `2>/dev/null` filter for warnings or note in skill that version warnings are informational.

### F3: `snow stage copy` path is relative but skill says "paths are relative to repo root"
- **Step:** 4 - Upload configuration files
- **Severity:** Low
- **Description:** The skill says to run from repo root with relative paths like `.cortex/skills/build-routing-solution/native_app/...` which works fine but the `cd` in Step 5 to `.cortex/skills/build-routing-solution/` changes the working directory context. Need to be careful about where you are.
- **Suggestion:** Make the working directory requirement explicit at the top of each step, or always use absolute paths.

### F4: Background build process monitoring difficult
- **Step:** 5 - Build and push container images
- **Severity:** Medium
- **Description:** Running all 4 builds+pushes in a single chained command means you can't see intermediate progress. If one fails midway, you waste time waiting. Docker push taking 5+ minutes for large images (openrouteservice is ~800MB) makes this worse.
- **Suggestion:** Consider building images in parallel where possible, or providing a progress monitoring script.

### F5: Docker push to SPCS registry extremely slow (~15+ min for 238MB image)
- **Step:** 5 - Build and push container images
- **Severity:** High (blocks entire workflow)
- **Description:** The openrouteservice image (238MB compressed) took 15+ minutes to push to the SPCS registry with no completion. The push process (PID 7249) was alive and consuming CPU but docker's carriage-return progress output was invisible via `tail -5`. The agent had no way to determine if the push was progressing, stuck, or failing. This blocked the entire workflow.
- **Root cause:** Network upload speed to Snowflake's SPCS registry + invisible progress output.
- **Suggestion:** (1) Add `--quiet` flag to docker push for simpler output, (2) add a timeout/retry mechanism, (3) add a step to verify each push individually with `snow spcs image-repository list-images` after each push rather than chaining all 4 in one command.

### F6: build-images.md uses `$CONTAINER_CMD` variable but agent bash calls don't persist variables
- **Step:** 5 - Build and push container images
- **Severity:** Medium
- **Description:** The reference uses `$CONTAINER_CMD` throughout but the skill correctly notes "each bash tool call starts a fresh shell." However, the agent still needs to manually inline `CONTAINER_CMD=docker` or chain everything. The reference code snippets use `$CONTAINER_CMD` which won't work if copy-pasted into separate bash calls.
- **Suggestion:** Hardcode `docker` or `podman` in the reference examples with a note to swap, or use a script that sets it.

### F7: build-images.md reference has the agent `cd` into skill directory but Step 4 runs from repo root
- **Step:** 4→5 transition
- **Severity:** Low
- **Description:** Step 4 uploads files using paths relative to repo root. Step 5's build-images.md says "All paths below are relative to the skill directory (`.cortex/skills/build-routing-solution/`)." This requires the agent to `cd` between steps, which is error-prone in stateless bash.
- **Suggestion:** Standardize on repo root as working directory for all steps, or use absolute paths throughout.

## Friction Points (Cleanup)

### F8: Cleanup skill discovery queries assume all databases exist
- **Step:** Cleanup Step 2h-2q
- **Severity:** Low
- **Description:** Discovery queries like `SHOW SCHEMAS IN DATABASE FLEET_INTELLIGENCE` and `SHOW TASKS IN SCHEMA FLEET_INTELLIGENCE.DWELL_ANALYSIS` will fail with "does not exist" errors if those databases/schemas were never created (as in a partial build). The skill should handle this gracefully.
- **Suggestion:** Wrap each SHOW in a TRY/CATCH or check database existence first before running schema-level queries.

### F9: Dropping ROUTING_ANALYTICS warehouse removes active session warehouse
- **Step:** Cleanup Phase 6
- **Severity:** Medium
- **Description:** After dropping ROUTING_ANALYTICS, the subsequent verification queries failed with "No active warehouse selected" because the session was using that warehouse. Had to explicitly `USE WAREHOUSE MY_WH` to continue.
- **Suggestion:** Add a step at the start of cleanup: `USE WAREHOUSE MY_WH;` (or whatever the user's default warehouse is) so dropping ROUTING_ANALYTICS doesn't break the session.

### F10: 90+ local Docker images accumulated from past builds
- **Step:** Post-cleanup observation
- **Severity:** Low (cosmetic, but wastes disk)
- **Description:** `docker images | grep pm-fleet-test` shows 90+ images from v1.0.5 through v1.0.95 of ors_control_app alone, plus 26+ demo-dashboard versions and multiple routing_reverse_proxy versions. This is ~12+ GB of local disk.
- **Suggestion:** Add a local cleanup command to the cleanup skill: `docker rmi $(docker images | grep pm-fleet-test | awk '{print $3}') 2>/dev/null` or document periodic local image pruning.
