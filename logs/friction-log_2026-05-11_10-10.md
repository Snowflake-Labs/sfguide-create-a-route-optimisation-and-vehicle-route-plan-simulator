## Friction Points

### F1. ors_control_app YAML tag mismatched image-versions.env
- **What happened:** `openrouteservice_app/services/ors_control_app/ors_control_app_service.yaml` declared `ors_control_app:v1.0.147` while the env file `image-versions.env` (single-source-of-truth) declared `ORS_CONTROL_APP_TAG=v1.0.141`. Following the skill's pre-deployment check rule, I built v1.0.141 (matching env) but the staged YAML pointed to v1.0.147.
- **Resolution this run:** Edited the YAML to v1.0.141 and re-uploaded to `@ORS_SPCS_STAGE/services/ors_control_app/`. CREATE SERVICE then succeeded.
- **Recommendation:** (a) Add a CI/pre-commit check (`scripts/check_image_versions.sh` already exists per build-images.md) to fail if any service YAML's image tag drifts from `image-versions.env`. (b) Or have `deploy.sh` (and the build-routing-solution skill) auto-template the service YAML from `image-versions.env` at upload time, rather than committing a YAML with a hard-coded tag.

### F2. Existing services on a "fresh" account
- **What happened:** When running module `01_core_infra.sql`, all 5 `CREATE SERVICE IF NOT EXISTS` statements reported "<NAME> already exists, statement succeeded" even though this run claimed a fresh install (services were not visible in earlier `SHOW SERVICES` until after module 01 ran). This was non-blocking because of `IF NOT EXISTS`.
- **Resolution:** Continued — `IF NOT EXISTS` is idempotent and the services subsequently came up RUNNING.
- **Recommendation:** The skill text "Fresh install assumed" understates the case where the workflow is re-run after a prior partial install (objects from a prior run can persist). Add a one-line note in Step 6: "If services already exist from a prior run, IF NOT EXISTS will keep the existing definition — to force a refresh, run `ALTER SERVICE ... SUSPEND` then `ALTER SERVICE ... FROM @stage SPECIFICATION_FILE=...` then `RESUME` per AGENTS.md."

### F3. Build commands' tail-pipe swallows push completion lines
- **What happened:** `docker push ... 2>&1 | tail -3` and `... | tail -5` truncated the final "<digest>: digest:..." line in some stages so it was unclear from console output whether a push fully completed. I had to run `snow spcs image-repository list-images` separately to verify.
- **Resolution:** Verified via list-images.
- **Recommendation:** The build-images.md reference already calls this out ("push progress uses `\r` updates"). Consider explicitly recommending `docker push $URL > /tmp/push.log 2>&1 && tail -5 /tmp/push.log` in the agent code blocks, instead of inline `| tail -5`.

### F4. ARM Mac control-app build flow requires manual swap
- **What happened:** On macOS arm64 with Docker, the multi-stage `Dockerfile.runtime` would crash inside QEMU during esbuild. Workaround per `build-images.md`: build `dist/` and `dist-server/` locally, then swap `.dockerignore` ↔ `.dockerignore.prebuilt` around the docker build. This worked but is brittle (interruption mid-swap leaves the wrong `.dockerignore` checked in).
- **Resolution:** Used the documented swap pattern; restored after build.
- **Recommendation:** Provide a small helper script `scripts/build-control-app.sh` that does `cp/build/mv` atomically with a trap to restore `.dockerignore` on any error/exit. Reference it from the SKILL build-images.md instead of the inline cp commands.

### F5. `snow sql -f` long-running scripts return no progress signal
- **What happened:** `dwell-analysis/references/sql-pipeline.sql` (creates 8 Dynamic Tables) ran ~5 minutes silently with no incremental progress shown to the agent. I had to repeatedly poll `pgrep` + `ls -la /tmp/<log>` to gauge progress.
- **Resolution:** Polled until exit.
- **Recommendation:** For DT-heavy pipelines, consider splitting the file or having the agent loop `snow sql -q "SHOW DYNAMIC TABLES IN SCHEMA ... ;" -c <connection>` as a heartbeat. Or simply state in SKILL.md: "Step 8 dwell pipeline: ~5 min, expect silence."

### F6. Tail of long SQL files surfaces the SELECT statement, not the SELECT result
- **What happened:** Several `tail -3 /tmp/d_*.log` ended at the trailing SELECT verification statement text (because `snow sql -f` echoes statement before result). Required `tail -15`+ to see the actual numeric verification table.
- **Resolution:** Increased tail size.
- **Recommendation:** Skill steps that say "verify counts" should show the expected default output column header (e.g., the verification SELECT in `seed-data.sql`) so the agent knows how many lines to read.
