## Friction Points

### F1: Podman machine not running despite `podman info` succeeding

- **Step:** Step 2: Detect runtime
- **Severity:** Medium
- **What happened:** `podman --version` returned `5.7.1` and `podman info` returned version metadata successfully, but `podman login` failed with "Cannot connect to Podman... connection refused". The Podman machine was not actually started — `podman info` on macOS can return local client info without the VM running, but actual container operations require the machine.
- **Resolution:** Ran `podman machine start` which took ~15 seconds.
- **Recommendation:** In Step 2, change the verification command from `podman info` to `podman machine ls --format '{{.Running}}'` or `podman system connection list` to check if the machine is actually running. Or simply always run `podman machine start` proactively (it's idempotent and fast if already running).

### F2: ARM Mac esbuild crash in Dockerfile.runtime multi-stage build

- **Step:** Step 5: Build and push container images
- **Severity:** High
- **What happened:** The `Dockerfile.runtime` is a multi-stage build where Stage 1 runs `npm run build` inside the container. On ARM Mac with `--platform linux/amd64`, esbuild crashes with a QEMU segfault: `The service was stopped` / `exit status 1`. The troubleshooting.md states: "The multi-stage Dockerfile.runtime will detect the pre-existing dist/ in the builder stage (npm run build becomes a no-op since output already exists)" — this is incorrect. The Dockerfile runs `COPY . .` then `RUN npm run build` which always re-runs esbuild regardless of existing dist/ content. The vite build does not check for existing output.
- **Resolution:** Created an inline runtime-only Dockerfile via heredoc that skips Stage 1 entirely: `FROM node:20-alpine` -> `COPY dist ./dist` -> `COPY dist-server ./dist-server` -> `COPY package.json package-lock.json ./` -> `RUN npm ci --omit=dev --legacy-peer-deps`. This worked perfectly.
- **Recommendation:** Fix `references/troubleshooting.md` and `references/build-images.md` to be accurate about the ARM Mac workaround. Either: (a) Create a dedicated `Dockerfile.prebuilt` that only does the runtime stage (COPY dist + npm ci --omit=dev), or (b) Modify `Dockerfile.runtime` to conditionally skip the build if dist/ already exists (e.g., `RUN [ -d dist ] || npm run build`). The current docs say "npm run build becomes a no-op" which is false — vite always rebuilds.

### F3: `podman push` output invisible through `tail`

- **Step:** Step 5: Build and push container images
- **Severity:** Low
- **What happened:** Podman push progress output uses carriage returns (`\r`) for in-place updates. When piped through `tail -N`, all progress lines are invisible (they overwrite each other on the same line, and `tail` only sees the final state). This made it impossible to track push progress and difficult to know when pushes completed.
- **Resolution:** Re-ran pushes without `tail` filter, and polled the image repository list via `snow spcs image-repository list-images` to verify completion.
- **Recommendation:** In `references/build-images.md`, add a note: "Push progress is invisible when piped through tail. Use `snow spcs image-repository list-images` to verify pushes completed, or redirect stderr to a file: `podman push $URL 2>push.log`."
