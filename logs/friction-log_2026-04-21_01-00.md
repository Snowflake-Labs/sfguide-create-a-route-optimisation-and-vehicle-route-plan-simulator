## Friction Points

### F1: Snow CLI OAuth authentication timeout

- **Step:** Step 3 (initial snow CLI usage)
- **Severity:** High
- **What happened:** The default OAuth authorization code flow (`authenticator = "oauth_authorization_code"`) in the `[ukb96706]` connection timed out waiting for browser-based authorization. This blocked all `snow` CLI commands.
- **Resolution:** Generated an RSA 2048-bit key pair (`/tmp/rsa_key.p8`), set the public key on user PRESZKE via `ALTER USER ... SET RSA_PUBLIC_KEY=...`, and created a new `[routing]` connection in `connections.toml` with `authenticator = "SNOWFLAKE_JWT"`.
- **Recommendation:** The SKILL.md should document that key-pair auth may be needed as a fallback. Add a "Connection Troubleshooting" section or a pre-flight check that tests `snow connection test -c <name>` before proceeding. Consider defaulting to key-pair auth in the skill instructions.

---

### F2: Docker `--ignorefile` not supported on Docker Desktop for Mac

- **Step:** Step 5 (Build images - ors_control_app)
- **Severity:** Medium
- **What happened:** The `build-images.md` references `--ignorefile .dockerignore.prebuilt` for the ARM Mac pre-built workaround, but Docker Desktop does not support the `--ignorefile` flag (it's a BuildKit-only flag not exposed in Docker CLI).
- **Resolution:** Used the `.dockerignore` swap technique: `cp .dockerignore .dockerignore.bak && cp .dockerignore.prebuilt .dockerignore` before build, then `mv .dockerignore.bak .dockerignore` after. This is documented in `build-images.md` as the alternative.
- **Recommendation:** Make the swap technique the PRIMARY documented approach since `--ignorefile` doesn't work on most Docker Desktop installations. Move `--ignorefile` to a footnote.

---

### F3: First Docker push fails with "Authorization Failure"

- **Step:** Step 5 (Push images)
- **Severity:** Medium
- **What happened:** The first `docker push` attempt failed with an authorization error even though `snow spcs image-registry login` had been run earlier. The token may have expired during the ~15 min image build phase.
- **Resolution:** Re-ran `snow spcs image-registry login -c routing` and retried pushes — all succeeded.
- **Recommendation:** Add a note in SKILL.md Step 5 to re-login to the registry immediately before pushing, especially after long builds. Consider adding `snow spcs image-registry login` as the first command in the push sequence.

---

### F4: Parallel SQL calls create schemas before databases exist

- **Step:** Step 3 (Setup DB/stages) and Step 8 (Deploy demos)
- **Severity:** Medium
- **What happened:** When running `CREATE DATABASE IF NOT EXISTS` and `CREATE SCHEMA IF NOT EXISTS` in parallel via `snowflake_sql_execute`, the schema creation sometimes executes before the database creation completes, causing "Database does not exist" errors.
- **Resolution:** Created databases first, waited for completion, then created schemas in a second batch.
- **Recommendation:** The SKILL.md should explicitly note that database creation must complete before schema creation. Add a comment in the SQL modules about execution ordering.

---

### F5: Docker push progress invisible with tail

- **Step:** Step 5 (Push images)
- **Severity:** Low
- **What happened:** Docker push uses carriage returns (`\r`) for progress updates, which `tail` cannot display. The output appears blank until push completes.
- **Resolution:** Waited for completion and verified via `snow spcs image-repository list-images`.
- **Recommendation:** Add a note that push progress is not visible in non-TTY environments. Suggest using `list-images` to verify after push.

---

- **Severity:** Low
- **What happened:** After editing `connections.toml` to add the `[routing]` connection, Snow CLI rejected the file because its permissions were too open.
- **Resolution:** `chmod 0600 ~/.snowflake/connections.toml`
- **Recommendation:** Note in SKILL.md that `connections.toml` requires restrictive permissions (0600).

### F8: Dwell Analysis sql-pipeline.sql overrides SLA thresholds from seed-data.sql

- **Step:** Step 8c (Dwell Analysis)
- **Severity:** Low
- **What happened:** The `seed-data.sql` inserts demo-friendly low SLA thresholds (WAREHOUSE: 5/15 min) with `WHERE NOT EXISTS` guard. But `sql-pipeline.sql` Step 4 also has SLA_THRESHOLDS with different values (WAREHOUSE: 60/120 min) and the same `WHERE NOT EXISTS` guard. Since seed-data.sql runs first, its values persist. This is actually the desired behavior (demo-tuned thresholds) but the two files having different values is confusing.
- **Resolution:** No action needed — seed-data.sql thresholds (lower values) remained because of the `WHERE NOT EXISTS` guard, which is correct for demo purposes.
- **Recommendation:** Add a comment in `sql-pipeline.sql` explaining that seed-data.sql intentionally uses lower thresholds for demo visibility, and the pipeline's higher thresholds are for production use.
