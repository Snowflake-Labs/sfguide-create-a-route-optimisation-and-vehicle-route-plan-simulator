## Friction Points

### 1. ors_control_app YAML referenced unbuilt image tag (v1.0.152) while `image-versions.env` had v1.0.143
**What happened:** `01_core_infra.sql` failed at the `CREATE SERVICE … ors_control_app` statement with `Image …/ors_control_app:v1.0.152 not found`. The skill's pre-deployment check note warns about this exact mismatch, but the env file and YAML are out of sync in the current branch.

**Resolution:** Re-tagged the locally built `ors_control_app:v1.0.143` image as `v1.0.152` and pushed:
```bash
docker tag $REPO_URL/ors_control_app:v1.0.143 $REPO_URL/ors_control_app:v1.0.152
docker push $REPO_URL/ors_control_app:v1.0.152
```
Then re-ran `01_core_infra.sql` (idempotent). All other services already existed and were skipped via `IF NOT EXISTS`.

**Recommendation:** Add a Step 4.5 (between upload & build) that diffs `openrouteservice_app/image-versions.env` against the image tag in `services/ors_control_app/ors_control_app_service.yaml`. If they disagree, either bump the env file or print a clear error before any image build starts. The skill already has a note describing this as a "Pre-deployment check" but it is buried in Step 6 — promote it to a hard validation at Step 5 entry.

### 2. `03_region_management.sql` final `BEGIN…ALTER TASK…END` block fails under `snow sql -f`
**What happened:** The trailing block
```sql
BEGIN
    ALTER TASK OPENROUTESERVICE_APP.CORE.RESCUE_PENDING_PROVISIONS_TASK RESUME;
EXCEPTION WHEN OTHER THEN NULL;
END;
```
errored with `syntax error … unexpected '<EOF>'`. The snow CLI splits scripts on `;` and gets confused by the `EXCEPTION` clause inside an unlabeled `BEGIN…END`.

**Resolution:** Manually issued `ALTER TASK IF EXISTS OPENROUTESERVICE_APP.CORE.RESCUE_PENDING_PROVISIONS_TASK RESUME;` via `snowflake_sql_execute`. Task now active.

**Recommendation:** Wrap the `BEGIN…END` in `EXECUTE IMMEDIATE $$ … $$`, or simply replace it with `ALTER TASK IF EXISTS … RESUME;` (which never errors on a missing task). Either change keeps the snow CLI happy.

### 3. `05_matrix_pipeline.sql` errors with "ambiguous column name 'ROAD_FILTER'" when re-run on already-deployed schema
**What happened:** The script issues `ALTER TABLE … ADD COLUMN IF NOT EXISTS ROAD_FILTER …`, but Snowflake returns `002028 ambiguous column name 'ROAD_FILTER'` when the column already exists. (This appears to be a Snowflake quirk — `IF NOT EXISTS` should be a no-op, not raise ambiguity.) snow CLI then aborts the script and the remaining matrix procedures are not re-applied.

**Resolution:** Verified all 11 matrix procedures + 4 column ALTERs already existed from a prior run, so module was effectively idempotent and could be skipped. Manually verified via `INFORMATION_SCHEMA.PROCEDURES` and `INFORMATION_SCHEMA.COLUMNS`.

**Recommendation:** Replace each `ALTER TABLE … ADD COLUMN IF NOT EXISTS X …` with a guarded EXECUTE IMMEDIATE that probes `INFORMATION_SCHEMA.COLUMNS` first, or move all columns into the original `CREATE TABLE IF NOT EXISTS` and drop the ALTERs. This makes Step 6 fully replayable.

### 4. `retail-catchment/seed-data.sql` REGIONAL_ADDRESSES INSERT silently terminated mid-script
**What happened:** First run terminated after creating CITIES_BY_STATE (133 rows) and RETAIL_POIS (48,102 rows). REGIONAL_ADDRESSES was created but had 0 rows; REGION_CONFIG did not exist. The 2.8M-row INSERT against `OVERTURE_MAPS__ADDRESSES` apparently exceeded a snow CLI / network timeout on a MEDIUM warehouse. No explicit error in stdout — the process simply died.

**Resolution:** Bumped `ROUTING_ANALYTICS` to LARGE, ran the INSERT directly via `snowflake_sql_execute` with `timeout_seconds=1200` (2,827,959 rows in ~30s), then created `REGION_CONFIG` and inserted the SF row manually. Restored warehouse to MEDIUM.

**Recommendation:** Add a `ALTER WAREHOUSE ROUTING_ANALYTICS SET WAREHOUSE_SIZE='LARGE';` at the top of `retail-catchment/references/seed-data.sql` and restore to MEDIUM at the end. Or split the addresses INSERT into a separate file with explicit warehouse-size handling so a snow CLI retry is safe. The skill should also document expected runtime for this step (currently says "3-5 minutes" but the addresses load alone can be several minutes on MEDIUM).

### 5. `dwell-analysis/references/sql-pipeline.sql` only created 4 of 8 Dynamic Tables on first run
**What happened:** First execution stopped after creating DT_STATE_CHANGES, DT_DWELL_SESSIONS, DT_DWELL_ENRICHED, DT_H3_CONGESTION. Re-running the script created the remaining 4 DTs (DT_SLA_ALERTS, DT_FACILITY_UTILIZATION, DT_DRIVER_DWELL_SUMMARY, DT_DAILY_TRENDS) plus SLA_ALERT_LOG and the task. Likely cause: the first DT initialization blocked the connection long enough that snow CLI's batched output framing dropped subsequent statements.

**Resolution:** Re-ran `snow sql -f .cortex/skills/dwell-analysis/references/sql-pipeline.sql -c <connection>` once. All statements use `CREATE OR REPLACE` so the rerun was idempotent. Final counts: state_changes=472,869, dwell_sessions=12,570, sla_alerts=5,240, daily_trends=8, drivers=50, facilities=5,756.

**Recommendation:** Either (a) split the pipeline into per-DT files so partial failures are obvious, or (b) add a verification block at the end that `SELECT COUNT(*) FROM SHOW DYNAMIC TABLES` and asserts 8 rows. Currently a partial run produces no error and leaves an obviously broken pipeline.

### 6. (Cosmetic) `routing-agent/references/deploy-agent.sql` final SELECT runs without a current database
**What happened:** Last verification SELECT in the script ran with no `USE DATABASE`, raising `Cannot perform SELECT. This session does not have a current database.` All 4 objects (3 procs + 1 agent) were created successfully; only the verify step failed.

**Resolution:** Confirmed objects via `SHOW PROCEDURES`. No actual deployment failure.

**Recommendation:** Prefix the verify SELECT with `USE DATABASE FLEET_INTELLIGENCE;` or fully-qualify the `INFORMATION_SCHEMA` reference (`FLEET_INTELLIGENCE.INFORMATION_SCHEMA.PROCEDURES`).
