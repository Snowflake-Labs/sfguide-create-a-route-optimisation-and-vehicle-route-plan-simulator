## Friction Points

### Friction #1 — Module `03_region_management.sql` contains an unwrapped `BEGIN…END` block that breaks `snow sql -f` parsing

**Symptom:** Line 320–325 of `03_region_management.sql` contains a bare `BEGIN ... ALTER TABLE IF EXISTS ... ADD COLUMN IF NOT EXISTS ... EXCEPTION WHEN OTHER THEN NULL; END;` block. The `snow` CLI splits on `;` and sends incomplete fragments to Snowflake, producing:

```
001003 (42000): SQL compilation error: syntax error line 3 at position 66 unexpected '<EOF>'.
```

This halts execution of the remaining ~13 procedures/functions in the file (BUILD_ORS_SERVICE_SPEC, CREATE_REGION_ORS_SERVICE, WRITE_ORS_CONFIG, RESUME_REGION_ORS, DROP_REGION_ORS, LIST_REGIONS, etc.), which are all required for region management and full control-app functionality.

**Resolution this run:** Extracted lines 327+ of the file into `/tmp/03_rest.sql` via `awk 'NR>=327'` and re-ran it with `snow sql -f`. All remaining procedures created successfully. The skipped `ALTER TABLE` block is a no-op migration anyway — the new `REGION_ORS_MAP` CREATE TABLE already includes `COMPUTE_SIZE` from the start (verified via DESCRIBE TABLE).

**Recommendation:**
1. Wrap the migration block in `EXECUTE IMMEDIATE $$ BEGIN ... END; $$;` so the `;` inside the block is not interpreted by the CLI splitter.
2. OR delete the block entirely since it is now redundant (the CREATE TABLE above it already defines COMPUTE_SIZE).
3. Add a verification step at the end of the skill workflow that lists the 13 procedures from module 03 and flags missing ones.

### Friction #2 — `snow spcs image-repository url` output wraps across terminal columns

**Symptom:** The raw stdout from `snow spcs image-repository url OPENROUTESERVICE_APP.core.image_repository -c <connection>` visually wraps the URL across two lines in the tool output (character 80 boundary). Storing via `$()` substitution is fine, but dumping via `cat file` shows what appears to be a newline-containing string.

**Resolution this run:** Used `--format JSON` with `python3 -c "import json; print(json.load(sys.stdin)['message'])"` to get a clean single-line value, then stripped whitespace to be safe.

**Recommendation:** Update `references/build-images.md` Step 2 to recommend `--format JSON` extraction to avoid confusion when debugging. Example:
```bash
REPO_URL=$(snow spcs image-repository url OPENROUTESERVICE_APP.core.image_repository -c <connection> --format JSON | python3 -c "import sys, json; print(json.load(sys.stdin)['message'])")
```

### Friction #3 — Image version drift between `image-versions.env` and `build-images.md`

**Symptom:** `references/build-images.md` Section 3 lists `ors_control_app:v1.0.119` and the inventory table lists `v1.0.117`, but `openrouteservice_app/image-versions.env` has `ORS_CONTROL_APP_TAG=v1.0.130`. The service YAML also references `v1.0.130`. Agents who follow the doc literally would build the wrong tag.

**Resolution this run:** Followed the pre-build instruction to treat `image-versions.env` as the source of truth and used `v1.0.130`.

**Recommendation:** Run a pre-commit hook (or add a CI check) that diffs tag values across `image-versions.env`, all `*_service.yaml` files, and `references/build-images.md`. Alternatively, replace the hardcoded tags in `build-images.md` with a note to read them from `image-versions.env` at build time.
